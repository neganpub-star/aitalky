## 六、深度审计:计费 + 会话 两高风险域(2026-06-26 补)

### 6.1 计费域 ✅ 扎实,无阻断问题
- **回调入账**(`BillingWalletServiceImpl.handleCoinlyCallback`):验签 → status=1 校验 → txid 幂等 → 地址 hash 反查项目 → 项目锁 `lock:bil:wallet:{pid}` → 事务内 version 乐观锁原子自增 + 行级 HMAC 防改库。四重防护到位。
- **核销**(`doPay`):订单状态校验 → **订单签名防篡改**(覆盖 amount/seats/quantity) → 余额校验 + version 乐观锁原子扣减 → markPaid 状态条件更新 → 开通订阅。全程钱包锁 + 事务。
- **自动核销**(`autoSettlePendingOrder`):重入钱包锁、状态条件幂等、catch BizException 不破坏回调 ack。
- **验签**(`CoinlySigner`):符合 Coinly 文档规则,拒空、重算比对。`equalsIgnoreCase` 非时序安全但对回调签名场景影响可忽略(低)。
- 金额全程 `BigDecimal`;`stripTrailingZeros` 标准化签名原文。
- 小瑕疵(低):回调 `new BigDecimal(amount)` 若 amount 非法会抛 NumberFormatException(非 BizException)→ 500;回调场景可接受。

### 6.2 会话域:多租户隔离 ✅ 安全(拦截器兜底)
- `ConversationServiceImpl.getById` 用 `selectById` 无显式 project 校验,但**多租户拦截器**(`MyBatisPlusConfig`)对 `cnv_conversation`(非忽略表、非 pf_ 前缀)在坐席有 TenantContext 时自动拼 `project_id=当前项目` → 跨项目 id 返回 null/NOT_FOUND。**跨项目串数据被拦截器兜住,无漏洞**。

### 6.3 🔴【高】消息 seq 计数器无 DB 兜底初始化(可靠性/数据完整性)
- **位置**:`MessageServiceImpl.send()` 第 41 行 `redisson.getAtomicLong("conv:seq:"+id).incrementAndGet()`。
- **问题**:seq 直接从 Redis 自增,**不从 DB 兜底**。一旦 `conv:seq:{id}` 丢失(Redis 重启无持久化 / key 被 maxmemory 驱逐 / 换新环境),计数器归零 → 新消息 seq 从 1 重发,**与已落库旧消息 seq 碰撞**。
- **后果**:会话 `last_seq` 倒退;前端按 seq 去重会丢新消息或乱序;直接**击穿项目铁律「seq=真相」**。
- **触发条件**:Redis 在已有消息数据后被清空/驱逐(prod 长期运行风险高;测试环境若用全新 DB+Redis 不触发)。
- **建议修复**(取/建计数器时,key 不存在则用会话 `lastSeq` 播种;CAS 播种避免并发重播):
  ```java
  var counter = redisson.getAtomicLong("conv:seq:" + convId);
  if (!counter.isExists()) {
      long dbLast = 会话.getLastSeq()（或 Mongo 该会话 max(seq)）;
      counter.compareAndSet(0, dbLast);   // 仅当仍为初始 0 时播种,幂等
  }
  long seq = counter.incrementAndGet();
  ```
- **待确认**:是否本轮修(我可改,核心热路径,要你点头)。**正式环境上线前必须修**;纯测试(全新库)可暂缓。
