# aitalky V1 后端模块设计

> 技术栈:Java 21 + Spring Boot 3 + Maven + MyBatis-Plus;MySQL+Mongo+Redis+RocketMQ+MinIO;Netty WS。
> 架构:**模块化单体**(DDD 模块,不上微服务)+ **多部署单元**(让坐席端业务/实时网关能各自横向扩展)。
> 核心约束:**坐席端业务无状态,用户量大时多实例横向扩展**。
> 整理时间:2026-06-08。

---

## 一、部署单元(3 个,各自可横向扩展)

```
                         ┌──────────────┐
   运营方 ───────────────►│ aitalky-admin│  平台后管API(少量运营,独立/内网)
                         └──────┬───────┘
                                │  共享业务模块 + 同一套 MySQL/Mongo/Redis
   租户/坐席(Web) ──LB──►┌──────┴───────┐
   后管(Web)      ──LB──►│ aitalky-app  │  租户/坐席业务API ★无状态,横向扩展 N 实例
                         └──────┬───────┘
                                │
   坐席端/客户端(WS) ─LB──►┌──────┴───────┐
                         │ aitalky-ws   │  实时网关(Netty)★横向扩展 N 实例 + Redis 路由
                         └──────────────┘
        共享: Redis(缓存/在线/WS路由/会话seq/分配/锁) · RocketMQ(异步+顺序) · MySQL · Mongo · MinIO
```

| 部署单元 | 职责 | 扩展性 |
|---|---|---|
| **aitalky-app**(主) | 租户/坐席/信使/会话/消息/团队/计费 的 REST API + 业务逻辑 | **无状态,LB 后挂 N 实例,随量横向扩** |
| **aitalky-ws** | 坐席端 + 客户端 的 WebSocket 长连接(Netty) | **横向扩 N 实例**,Redis 存"连接→实例"路由 |
| **aitalky-admin** | 平台运营后台 API(管全平台用户/订阅/订单/套餐/协议) | 运营用,量小;**独立部署=安全隔离**(god 权限不与租户API同进程) |

> 为什么这样拆而不是微服务:三者**共享同一套业务模块代码 + 同一套库**(不是按域拆服务、不搞服务间 RPC),只是**不同入口/不同扩展节奏**。app 要随用户量扩、ws 连接特性不同要单独扩、admin 权限敏感要隔离——这是"模块化单体的多入口部署",保留单体的开发效率。

## 二、Maven 模块结构(代码)

```
aitalky/  (parent pom)
├── aitalky-common            # 通用: 统一响应/异常/雪花ID/工具/枚举
├── aitalky-framework         # 基建封装: MyBatis-Plus(多租户拦截器/审计填充)/Redis(Redisson)/RocketMQ/MinIO/安全(JWT)/**i18n(MessageSource,API提示按调用方语言返回)**
│
│   # ===== 业务模块(DDD 限界上下文, 纯库, 被入口装配) =====
├── aitalky-platform          # pf_  平台后管: admin/role/plan/plan_quota/addon_pack/agreement + 全平台用户/订阅/订单管理视图
├── aitalky-identity          # id_  身份租户: account/project/role/member/invite/invite_link + 鉴权/RBAC/多租户上下文
├── aitalky-messenger         # mse_ 信使: messenger/i18n/api管理(appId/secret)/语言库
├── aitalky-routing           # asn_ 会话设置/客服组/**分配引擎**(手动/轮流/负载+承载量+等待队列)
├── aitalky-customer          # cus_ 客户: 用户/游客身份/聚合/画像/自定义属性
├── aitalky-conversation      # cnv_ 会话: 生命周期/收件箱视图/分配·转派/保持期自动结束
├── aitalky-message           # 消息(Mongo): 收发/富消息/翻译/已读/撤回/**序号+可靠投递**
├── aitalky-billing           # pay_+pf_plan: 订阅/配额(quota)/加密支付(NexaPay 对接)/订单
│
│   # ===== 部署入口(thin, 装配业务模块 → 可执行 jar) =====
├── aitalky-app               # 装配 identity+messenger+routing+customer+conversation+message+billing → 业务API(无状态)
├── aitalky-admin             # 装配 platform(+只读引用其他模块的查询) → 平台后管API
└── aitalky-ws                # Netty WS 网关(依赖 message/conversation/identity 的服务做鉴权与推送)
```

依赖方向:`入口 → 业务模块 → framework → common`;业务模块间**只依赖对方的接口(api 包),不碰对方的 Mapper/表**。

## 三、横向扩展关键设计(重点)

### 3.1 业务 app 无状态
- **JWT 鉴权**(不存服务端 session)→ 任意实例处理任意请求。
- **不在 JVM 内存存业务状态**:在线状态/路由/缓存/分配队列/会话 seq **全放 Redis**。
- → app 实例可随意增减,LB(Nginx / K8s Service)轮询即可,**随用户量横向扩**。

### 3.2 WS 网关横向扩展 + 多端推送(关键)
- 每个 ws 实例只持有自己那批长连接;多实例水平扩。
- **一个成员/客户可有多条连接**(多浏览器标签、Web + App 多端)→ Redis 存:
  - `ws:conn:member:{memberId}` = Set{实例,connId}(该成员**全部**连接);客户同理 `ws:conn:cust:{customerId}`。
  - **在线状态 = 该 Set 是否非空(引用计数,最后一条连接断开才算离线)**。
- **不做单点登录顶号**:cicada 正是"单点登录 + 按 accountId 路由"导致**串窗口/漏推**;aitalky **允许多端并存,推送给该成员/客户的全部连接**。
- 推送:业务侧发 RocketMQ/Pub-Sub(带目标连接集合)→ 各 ws 实例消费、推给自己持有的目标连接。

### 3.3 并发安全(多实例下不出错)
- **会话分配**:多实例同时分配易重复 → 用 **Redis 分布式锁 / 原子操作**(按 conversationId/坐席),保证一个会话只分给一个坐席、不超承载量。
- **会话序号 seq**:`Redis INCR(conv:seq:{cid})` 原子生成,跨实例不重复(保序的基础)。
- **幂等**:消息按 msgId 去重;订单/支付回调幂等处理。

### 3.4 分布式定时任务(避免多实例重复跑)
- **保持期自动结束会话**、**订单支付超时关单**、**订阅到期处理** = 定时任务。
- 多实例下必须**只跑一次** → 用分布式调度(**XXL-Job** 或 Redisson 分布式锁/定时),不要用单机 `@Scheduled` 裸跑。

### 3.5 数据层扩展
- MySQL:读多可上**读写分离**(主写从读);租户量极大再分库。
- Mongo:消息按 `projectId+conversationId` **分片**横向扩。
- Redis:集群模式。
- 这些是"扩展余地",V1 单实例起步,架构不挡路。

### 3.6 推送目标:多端 + 代看/代发(直击 cicada 痛点)
一条会话 C 的新消息/事件,推送给以下连接的**并集**:
1. **会话负责人(assignee)的全部连接**(多端都收);
2. **当前正在查看 C 的所有连接(会话订阅者)**——客户端打开某会话即向 ws **订阅**该会话(`ws:sub:{conversationId}` = 连接集合),关闭即退订。

由此**天然解决** cicada 的几个坑(不再打补丁):
- **负责人代看 C**:其连接订阅了 C → 自动收到推送(取代 cicada 的"按 supervisor 补推")。
- **负责人代下面坐席回消息(代发)**:消息归 assignee 坐席 → 推给 **assignee 全部连接 + 全部订阅者**(含坐席本人各端 + 代发的负责人),客户端按 `msgId` 去重。**坐席账号和负责人账号都能收到** ✅(正是你说的 cicada 那个场景)。
- **客户端多端**:推给该客户全部连接。
> 一句话:用 **多连接 + 会话订阅 + 推全部** 取代 cicada 的 **单点登录 + 按 accountId 补推**,从根上保证"多端、代看、代发都收到,且不串不丢"。

### 3.7 后端多语言(API 提示)
- aitalky-app **返回给前端的提示/错误信息要多语言**(不是只前端 i18n)。
- 实现:Spring **MessageSource** + `messages_zh_CN.properties` / `messages_en_US.properties` …;**错误码 + i18n 文案**。
- 语言来源:请求头(`Accept-Language` 或自定义 `lang`)或当前坐席的 `id_member.language`;统一响应包装时按该语言取文案。
- 信使端(客户)接口的提示按 URL `lang` 返回。
