# aitalky V1 技术选型（行业最佳实践方案）

> 系统类型：多租户 SaaS 客服 IM 平台（Intercom / Chatwoot 同类）。核心诉求：实时消息、多租户隔离、可水平扩展、干净可维护、为 V2(AI/知识库/agent) 预留。
> 原则：按**行业最佳实践**给主推方案；同时标注「团队适配」(你们 Java/Vue 强、5 人)与「与 cicada 差异」。
> 整理时间：2026-06-08。

---

## 一、架构风格：模块化单体（Modular Monolith）

- **推荐**：单体应用内按 DDD 划清模块边界（身份/信使/会话/消息/订阅/智能…），模块间走接口/事件，**不共享对方的库表**。
- **为什么是最佳**：5 人团队 V1 上微服务是典型过度设计（分布式事务、服务治理、联调成本爆炸）。模块化单体 = 微服务的边界纪律 + 单体的开发效率，**将来某模块真成瓶颈再独立拆出**（边界清晰，拆起来低成本）。
- **唯一例外**：WS 实时网关建议**从一开始就独立成一个可单独扩展的进程**（见第四节），因为它的扩展特性和业务服务不同。

## 二、后端：Java 21 + Spring Boot 3.x

- **推荐（已定）**：**Java 21 + Spring Boot 3.x + Maven**。
- **为什么 Java 21**：只选 **LTS(长期支持)** 版(17/21/25)，非 LTS(22/23/24)仅 6 个月就 EOL，生产不可用。21 自带 **虚拟线程(Virtual Threads)**——对"大量长连接+阻塞IO"的 IM 服务极契合(几十万并发不爆)，且生态/库/中间件全面支持、踩坑最少。(求新可上 Java 25 LTS，但需确认第三方库适配。)
- **为什么 Maven**：团队熟(cicada 在用)、Spring 生态默认、零学习成本；构建工具不影响架构，稳妥优先。(Gradle 仅在追求构建速度时才考虑。)
- **行业现状**：客服/IM 后端业界有 Java(Netty)、Go、Node、Elixir(Phoenix) 几派。**Elixir/Go 在"海量长连接"上更极致**，但你们 5 人是 Java 团队——换语言是纯风险。**Java + Netty 扛 10 万级长连接没问题**(微信早期、众多 IM 都是 Java/Netty)，是"团队适配下的行业最佳"。
- **持久层**：**MyBatis-Plus**（团队熟、SQL 可控；客服分配/会话聚合查询复杂，ORM 自动生成的 SQL 反而难调优）。简单 CRUD 用 MP，复杂查询写 XML 或引入 **jOOQ**(类型安全 SQL)。
  > 备选：Spring Data JPA(开发快但复杂查询吃力)。
- **鉴权**：**Spring Security + JWT**，token 里带 `tenantId/projectId/roles`，配合 RBAC。
  > 备选：Sa-Token(更轻、国内常用、cicada 在用)——若想快可选它。
- **API**：REST + **OpenAPI(springdoc)** 自动文档；对外/对接用 **Webhook**。

## 三、数据库：MySQL 8（业务）+ MongoDB（消息）✅（已定）

- **职责划分**：
  - **MySQL**：项目/账号/成员/角色权限/**会话(元数据+分配)**/订阅/设置/**会话维护(收件箱列表用)**——关系型 + 事务。
  - **MongoDB**：**消息体**(海量、追加写、按会话读)；富消息/Web3 自定义属性用文档结构。
- **为什么这套（团队适配下的稳健选择）**：团队最熟、可直接复用 cicada 经验；Mongo 天生适合海量消息(高写入吞吐 + **分片横向扩展**)。

### 性能要点（澄清"消息量大也快"）
- **总量大 ≠ 读得慢**：读永远按单会话(`where conversationId=X order by timestamp desc limit N`)，走 `(conversationId, timestamp)` 索引，**B 树 O(log n)**——总量 100 万行还是 10 亿行，读单会话速度几乎一样。**总消息量到亿级很常见，但不影响按会话读取**。
- **真正撑不住单实例的是**：① 写入并发 QPS；② 单机数据体量/IO；③ 超大单表的运维(备份/迁移/建索引)。
- **应对**：到达上限时 **Mongo 分片(sharding)** 横向扩到多机，写/存摊开，**读仍快**(分片键+索引)。这正是选 Mongo 的价值——上限高。

### 三条纪律（避开 cicada 的坑）
1. **边界清晰、单一真相**：MySQL 管业务、Mongo 管消息体，**同一字段只存一处**(cicada 的 bug 多源于 nickName 等多处不一致)。
2. **Mongo 索引/分片键**：按 `conversationId`(或 `projectId+conversationId`)，保证按会话读快、将来好分片。
3. **收件箱列表走 MySQL**：MySQL 维护"会话"表(最后消息预览/时间/未读数)，新消息来时更新——**列表查询不扫 Mongo**；打开会话才去 Mongo 拉明细。

> V2 知识库 RAG 的向量检索：MySQL/Mongo 无原生向量，届时单独上向量库(**Milvus / Qdrant**)。V1 不涉及。
> 缓存/在线/路由仍用 Redis；文件用 MinIO。

## 四、实时通信：Netty WS 网关 + Redis 路由（最关键的一层）

- **架构(行业标准做法)**：
  1. **独立 WS 网关进程**(Netty)，只管长连接、心跳、收发帧；可**水平扩展多实例**。
  2. 连接是**有状态**的(某用户连在某台网关)。在 **Redis** 记录 `用户/会话 → 网关实例` 的路由 + 在线状态(presence)。
  3. 业务侧要推消息给某用户时：发到 **MQ/Redis Pub-Sub**，对应网关实例消费后 `writeAndFlush` 到那条连接。
  > cicada 已经是这套思路(Redis + RabbitMQ 路由)——保留，做干净。
- **协议**：WebSocket + JSON(或 Protobuf 提性能)。定义清晰的 cmd 协议。
- **为什么独立**：WS 网关的扩展/部署节奏和业务服务不同，独立后能单独扩容、单独发版不影响业务。

## 五、缓存 / 队列 / 存储 / 搜索

- **Redis 7 (+Redisson)**：在线状态、WS 路由表、会话分配队列、缓存、限流、分布式锁。行业标配。
- **RocketMQ**(一套):异步解耦(消息入站→处理→推送、翻译、通知)+ **顺序消息**(按 `conversationId` 保序)。团队熟(cicada 在用),合并掉 cicada 的 RabbitMQ+RocketMQ 双套。
  > 顺序消息只是"传输保序",真正的顺序/可靠投递保证靠**应用层会话序号 + 上线同步补推**(见《领域模型与表结构.md》第五节)。
  > V2 做"客户洞察/营销/事件分析"时,再评估引入 **Kafka** 做事件流/数仓入口(补充,不替换)。
- **MinIO**：图片/文件(S3 协议)，将来可无缝迁云对象存储。
- **搜索**：V1 用 **MySQL 全文索引(ngram 解析器支持中文)** + Mongo 文本检索；需求变重再上 **OpenSearch/Elasticsearch**。

## 六、前端（已定）

| 应用 | 技术 | 说明 |
|---|---|---|
| 坐席工作台 (web) | **React + TS + Vite + Ant Design** | V1 重点 |
| 后管 (web) | **React + TS + Vite + Ant Design** | V1 重点 |
| im-h5 (客户信使) | **React + Vite**(可用 **Preact** 减小体积) | 全 Web 统一 React,重写;单页聊天 |
| 坐席 App (iOS/Android) | **uni-app**(Vue 系,可顺带小程序) | **V1 后置开发**,先做 Web |
| 共享 | **pnpm monorepo** + 纯 TS 包(`api-sdk`/`ws-client`/`types`) | 各端**逻辑层复用** |

- **Web(三个)全用 React**：团队熟、生态/人才池最大、SaaS 行业标准、复杂实时 UI 库最全；统一一套 React 心智 + 共享组件/SDK,无 Vue 维护负担。
- **UI 库**：工作台/后管 **Ant Design**(数据密集后台最强、中文友好);im-h5 移动 H5 可用 **antd-mobile** 或自定义聊天 UI,并考虑 **Preact** 减小嵌入包体。
- **状态/请求**：Zustand(轻) 或 Redux Toolkit；数据请求 **TanStack Query**；路由 React Router。
- **im-h5 重写**：虽是单页,但含实时聊天复杂度(WS/重连、多类型消息、撤回/已读、历史分页、多语言、提醒、文件上传);WS/逻辑走共享 `ws-client`/`api-sdk`,React 重写约数天。
- **App 用 uni-app**：一套 Vue 语法出 iOS/Android/小程序;与 React Web UI 不共享,但**逻辑/类型经共享 TS 包复用**;**放在 Web 之后**。(将来若想全 React,可改 React Native。)

> **V1 前端范围 = Web 端(坐席工作台 + 后管 + im-h5),全 React**；坐席 App(uni-app)列入 V1.5/V2。

## 七、AI / 翻译

- **V1**：实时翻译，在 **Java 一个翻译模块**里封装翻译 API(复用 cicada 的 provider)。**预留「智能」边界上下文**。
- **V2(对标 ByteTrack 差异化)**：知识库 RAG、数字员工(Finn)、智能流程——**独立 Python(FastAPI) 服务**，向量库用 **Milvus / Qdrant**(MySQL/Mongo 无原生向量)，LLM 走 API；通过 REST/MQ 与 Java 核心通信。
- **为什么分语言**：LLM/agent 生态(LangChain/LlamaIndex 等)在 Python 最成熟；Java 核心不碰 LLM 细节，各自独立演进。

## 八、横切关注点

### 多租户隔离（SaaS 根基）
- **V1**：**共享库 + `project_id` 列**，所有查询**强制租户过滤**——用 **MyBatis 拦截器 / MP 多租户插件全局注入 `project_id`**(MySQL 无行级安全 RLS，必须靠拦截器全局生效，杜绝任何查询遗漏)。
- **独立部署客户**：同一套代码 → **每客户一个库/一套实例**(DB-per-tenant)。架构两者兼容。

### 可观测性（行业标准三件套）
- 指标 **Prometheus + Grafana**；日志 **Loki**(或 ELK)；链路追踪 **OpenTelemetry + Jaeger**；错误 **Sentry**。

### 部署 / DevOps
- **Docker** 容器化(Day 1) → **Kubernetes**(WS 网关/业务服务水平扩展)；起步可 Docker Compose。
- **CI/CD**：GitHub Actions(或 GitLab CI)，自动构建/测试/镜像/部署。
- 配置中心/密钥：环境变量 + (K8s ConfigMap/Secret 或 Nacos/Vault)。**杜绝 cicada 那种硬编码库名/IP**。

### 工程质量
- 测试：**JUnit5 + Testcontainers**(起真实 MySQL/MongoDB/Redis/RabbitMQ 跑集成测试)；前端 Vitest。
- 代码规范：Spotless/Checkstyle(后端) + ESLint/Prettier(前端)；统一 API 错误结构。

## 九、与 cicada 的主要差异（"行业最佳"带来的升级）

| 维度 | cicada | aitalky(本方案) |
|---|---|---|
| 架构 | RuoYi 重框架 + 业务散落 | 干净模块化单体 + DDD |
| 数据库 | MySQL+Mongo,但双写/字段不一致 | **同样 MySQL+Mongo,但边界清晰/单一真相/纪律严** |
| MQ | RabbitMQ + RocketMQ 双套 | **RocketMQ 一套**(顺序消息) |
| 多租户 | tenant_id 时有时无、硬编码库名 | **拦截器全局强制注入 project_id** |
| 前端 | Vue2+Vue3 混用 | **Web 三端(工作台+后管+im-h5)=React+AntD**;App=uni-app(后置);pnpm monorepo |
| 部署 | — | 容器化 + K8s + 可观测 |
