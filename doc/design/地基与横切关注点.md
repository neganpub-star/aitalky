## 一、后端地基

### 已打好 ✅
- 统一响应 `R` + 错误码 `ResultCode` + 全局异常 `GlobalExceptionHandler`
- 多租户隔离:`project_id` 由 MyBatis 拦截器自动注入(跨项目数据天然隔离)
- RBAC 功能权限:`@RequiresFunction` 注解 + 拦截器(接口级)
- JWT 无状态鉴权 + `TenantContext`(项目/成员/角色/functions/语言)
- 后端 i18n(MessageSource,按调用方语言返回提示)
- 雪花ID;审计字段自动填充(create/update_by/time);逻辑删除(del_flag)
- 分布式锁(Redisson)+ 分布式定时(锁保证只跑一次)
- Long→String 序列化(雪花ID防前端丢精度)
- 密码 RSA 传输 + 邮箱验证码(防刷)

### 建议现在补 🔶(后期补救最痛)
| 项 | 为什么现在做 | 方案 |
|---|---|---|
| **数据库迁移 Flyway** | 现在 26 表手工跑 sql;表会一直加,多人/多环境手工同步必乱。现在 baseline 最省事 | 引入 flyway,`V1__init.sql`(现有 ddl)+`V2__seed.sql`;以后每次改表加一个版本文件 |
| **操作/审计日志** | 删成员/改角色/支付等敏感操作必须留痕;后加要回头改一堆方法 | `@Log("...")` 注解 + AOP 切面 → 写 `sys_oper_log`(操作人/项目/动作/参数/IP/时间/结果) |
| **统一时间/时区** | 前后端时间格式不统一,后期一处处改格式很烦 | Jackson 全局:`LocalDateTime`↔`yyyy-MM-dd HH:mm:ss`(或统一时间戳ms);时区 Asia/Shanghai |
| **枚举规范** | 现在状态全是裸 `tinyint`(1/0/2),散落各处;后期改语义化枚举是大面积 churn | 关键状态用 Java enum + MyBatis-Plus `IEnum`;或至少集中常量类 |
| **接口限流/防刷** | 安全规范要求敏感接口限频;登录/发码/支付防爆破 | Redisson 限流 + `@RateLimit(key,count,period)` 注解,先搭框架按需贴 |
| **链路追踪 traceId** | 跨 app/ws 排错;后加要改日志格式 | Filter 注入 MDC traceId + 响应头返回;logback 输出含 traceId |
| **logback 配置** | 目前用默认 console;生产要文件滚动/分级 | logback-spring.xml:控制台+文件滚动,格式含 traceId/项目id |
| **文件上传(MinIO)** | 头像现在让填 URL;实际头像/图片消息要上传 | `FileService.upload()` 封装 MinIO,返回可访问 URL;统一大小/类型校验 |
| **信使端(客户)鉴权** | 会话模块的前置:客户**不能**用坐席 JWT;要独立的客户身份令牌(按 appId+userId/visitorId 签发) | 设计客户端 token 签发/校验(WS `resolveIdentity` 的 TODO);独立放行链路 |
| **逻辑删除 × 唯一索引** | `id_account.email`、`id_project.app_id` 唯一;逻辑删除后旧值仍占唯一键,再注册同邮箱会冲突 | 定策略:唯一键带 del_flag / 或删除时改写值 / 或物理删除。**现在定** |

## 四、决策记录(已落地)

### Flyway(已接入)
- 表结构以 `aitalky-app/src/main/resources/db/migration/V{n}__xxx.sql` 版本管理;**仅 app 入口执行**(admin/ws 不引 flyway)。
- 已有:V1 建表(26)/V2 种子/V3 sys_oper_log。**以后改表新增 V{n} 文件,不改旧文件**;启动自动迁移。
- `doc/design/ddl-mysql.sql` 仍为设计参考;可执行真相在 migration。

### 可观测性(已落地)
- 时间统一 `yyyy-MM-dd HH:mm:ss`(JacksonConfig);Long→字符串。
- traceId:`TraceIdFilter` 注入 MDC + 回传 `X-Trace-Id` 头;logback 输出含 traceId,控制台 + `logs/{app}.log` 滚动。
- 操作日志:`@Log("动作")` + `LogAspect` → `sys_oper_log`(操作人/IP/耗时/成功/失败信息);默认不记参数,`saveParams=true` 才记且禁含敏感字段。已贴成员管理写操作。

### 数据规范(已定)
- **枚举**:已开启 MyBatis-Plus `MybatisEnumTypeHandler`。**新字段的状态/类型优先用语义化枚举**(枚举实现 `@EnumValue` 存库 + `@JsonValue` 出参);存量 tinyint 在改到对应模块时顺手替换,不一次性大改。
- **逻辑删除 × 唯一索引策略**(现在定):
  - V1 **不物理删除** `id_account` / `id_project`(`email`/`app_id` 唯一键不会因逻辑删除冲突)。
  - **成员删除 = 逻辑删除**;重新邀请同账号时**复活软删记录**(`del_flag=0` + 重置角色)而非新插入 → 规避 `uk(project_id, account_id)` 冲突。该逻辑在「邀请」模块落地。
  - 通则:带唯一键的表若将来需要可物理删除的语义,用"复活"或物理删除,避免软删占用唯一键。

### 前端地基(本批进行中)
- react-i18next(文案走 `t()`)+ Zustand(登录态集中)+ ESLint/Prettier + 暗色主题(AntD darkAlgorithm + 主题切换)。
