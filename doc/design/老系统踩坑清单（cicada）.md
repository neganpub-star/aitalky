# 老系统（cicada）踩坑清单 —— aitalky 开发避坑指南

> **用途**:aitalky 参考 cicada 重写。cicada 在多坐席/多账号、负责人代看、WS 弱网补偿等场景踩过一连串坑(多数是上线后才暴露、靠"刷新才恢复"糊住的)。本清单把这些**实际翻车案例**收集归类,aitalky 开发对应模块前**必读**,从设计上一次规避,不要重蹈。
>
> 来源:`/Users/macintoshhd/work/customerService` 下 `cicada`(后端)、`cicada-web`(坐席台)、`im-h5`(客户端) 三仓库 2026-06 修复记录。
>
> 消息可靠性的**正向设计方案**见同目录《消息可靠投递与漏消息补偿.md》(seq 机制),本清单只补充 cicada 的**反面案例**佐证为什么那套方案是对的。

---

## 贯穿全篇的两条主线

cicada 这批坑几乎都源于两个被低估的维度:

1. **`accountId`(坐席)维度缺失** —— cicada 早期假设"一个客户 ↔ 一个会话窗口",大量逻辑只按 `customerId`/`chatId` 寻址。一旦**一个客户同时跟多个坐席聊**(尤其负责人代看多账号)就全面串台:消息串窗、会话被挪走、头像写串。
2. **WS 被当成消息流的唯一来源** —— 断连窗口期推送即弃、重连不对账,漏帧只能靠用户手动刷新。

aitalky 的铁律(CLAUDE.md 第 7、8 条)正是针对这两点。下面分主题展开。

---

## 一、消息可靠性 / 漏消息补偿(WS 断连弱网)

> 详细方案见《消息可靠投递与漏消息补偿.md》。这里是 cicada 的具体翻车点。

### 坑 1.1 断连窗口期的消息推完即弃,服务端不重推
- **现象**:息屏/切后台/冷热启动重连 WS,断连期间通过 WS 推的消息直接丢;消息其实已落 Mongo 没丢,但前端不补拉,非得手动刷新/重开会话才显示。
- **根因**:WS 推送是"尽力一次",无重连补偿动作。
- **aitalky 规避**:**WS 只当"通知",seq 当"真相"**。重连 `onopen` 自动 `sync(afterSeq)` 增量拉;列表/详情回带 `lastSeq` 对账。绝不把 WS 当成消息累积的唯一来源。
- 来源:cicada `8c666e85`(服务端加 `CUSTOMER_SYNC` 命令补拉)、im-h5 `2c6e9e3`、cicada-web `cce7a0c`。

### 坑 1.2 切前台补拉把新消息 unshift 到列表顶部 → 错位、看不到
- **现象**:回前台补拉时,把断连期漏的新消息插到列表**最旧位置**且不排序,用户在底部看不到;且只补第一页(10 条),超一页补不全。
- **根因**:补拉复用了"加载更早历史"的 `prepend` 逻辑,方向反了;分页也只取一页。
- **aitalky 规避**:**补回的消息按 seq 有序插入到正确位置**(不是无脑 prepend 也不是 append 到底);补拉要**逐页往前拉直到与本地衔接**,带页数上限(cicada 用 20 页保护),覆盖整个缺口。
- 来源:im-h5 `719703c`。

### 坑 1.3 ⚠ 补拉水位线用"本端发消息的本地时钟"做基准 → 漏拉
- **现象**:补拉基准取错,导致部分消息永远补不回来。
- **根因**:cicada 用 **timestamp** 做增量水位线,却拿"本端最后发出消息的本地时间"当基准——本地时钟与服务端有偏差,基准偏大就漏拉。
- **aitalky 规避**:**aitalky 用 `seq` 不用时间戳做水位**,从根上规避时钟问题。即便要用时间,也只能用**"最后一条收到的对方消息的服务端时间戳"**做基准,绝不用本端本地时钟。
- 来源:im-h5 `719703c`(水位线改取最后一条对方消息的服务端时间戳)。

### 坑 1.4 三路收消息(实时/WS补拉/前台补拉)去重不统一 → 重复或漏
- **现象**:同一条消息经不同路径进来,要么重复显示要么互相覆盖。
- **根因**:几条路径各用各的判重 key。
- **aitalky 规避**:**统一按服务端消息主键(seq / 服务端 _id)幂等去重**,三路共用一套 merge。客户本地 id 必须与服务端一致。
- 来源:im-h5 `719703c`(统一按服务端 `_id` 去重)。

### 坑 1.5 补拉补回来的消息不响铃 → 用户感知不到新消息
- **现象**:只有实时推送(cmd 20004)响铃,两条补拉路径只合并不提示,断连期漏的消息补回来悄无声息。
- **根因**:提示逻辑只挂在实时推送分支。
- **aitalky 规避**:补拉路径若补回**对方**新消息,也要走同一套提示(响铃/未读)。**一批补拉统一响一次**(不按条数刷屏),提示函数自带节流(cicada 用 500ms),多路同时触发只响一次。
- 来源:im-h5 `d4b2a42`。

## 二、多坐席 / 多账号会话隔离(accountId 维度) —— 重灾区

### 坑 2.1 一个客户同时跟多个坐席聊,实时消息/撤回串窗口
- **现象**:客户被多个坐席发消息时,实时消息和撤回因服务端只按 `customerId` 找连接,串到同一个窗口;刷新走 HTTP 历史才正常。
- **根因**:会话连接只按 `customerId` 寻址,丢了 `accountId`(坐席)维度。
- **aitalky 规避**:会话连接/推送**按 (客户 + 坐席) 隔离**。cicada 最终把会话槽 `customer:session:{customerId}` 改成 **Hash(field=accountId, value=sessionId)**:一个客户对每个坐席各占一个会话槽,可并存多窗口;reply/withdraw/已读回执**按 accountId 定向推**到对应坐席窗口,广播类才推给该客户所有窗口。
- **推送前校验**:推给客户前,校验客户连接 channel 上绑定的坐席 `ACCOUNT_ID` == 消息所属 `accountId`,不一致不推。
- 来源:cicada `30d424fe`(连接按客户+坐席隔离)、`ab9d1916`(推送前 accountId 校验)、`fc9c3f6a`(单点登录的演进)。

### 坑 2.2 前端"是否插入当前会话"只比 chatId 没比 accountId → 消息串窗
- **现象**:同一访客(chatId)同时跟多账号聊,发给**别的账号**的消息因 chatId 相同被错误插入当前打开的窗口,刷新才恢复。
- **根因**:前端 `insertMessage` 的"是否当前会话"判断只比了 `chatId`,漏了 `accountId`。
- **aitalky 规避**:前端所有"是否属于当前会话/当前账号"的判断,**必须同时比对会话标识和账号标识**,各分支保持一致(cicada 是某个分支漏了与其它分支不一致)。
- 来源:cicada-web `20bc1de`。

### 坑 2.3 入组事件(cmd20)不校验 accountId → 会话被"挪走"消失
- **现象**:同一访客跟多账号聊时,别的账号的入组事件按 chatId 把**当前账号的会话从分组里摘走**,表现为会话"消失"(实为被移动,刷新恢复)。
- **根因**:入组 mutation 只按 chatId 定位会话,跨账号误操作。
- **aitalky 规避**:入组/移动会话类事件,**入口门控 + mutation 内部 accountId 守卫双保险**;只处理属于当前正在看的账号的事件。
- 来源:cicada-web `84a84a0`(入口门控)、`dcc7661`(mutation 内部防御)。

### 坑 2.4 单点登录维度选错 → 多窗口互相干扰
- **演进**:cicada 先按 `(userId, customerId)` 判定单点登录 → 改成按 `customerId` 单点(同客户开新窗踢旧) → 最终为支持多坐席并存,改成**按 (客户, 坐席) 判定**(同一坐席重开才踢旧,不同坐席并存)。
- **aitalky 规避**:想清楚单点登录的**粒度**。若允许一个客户同时找多个坐席,单点必须落在 (客户+坐席) 粒度,不能一刀切按客户踢。
- 来源:cicada `fc9c3f6a` → `30d424fe`。

## 三、负责人代看 / 代管(customer_admin)

### 坑 3.1 代看时收不到实时更新,要刷新
- **现象**:负责人代看团队成员会话,新会话入组、翻译结果、会话设置变更等不实时,得刷新。
- **根因**:这些事件原本只推给会话归属成员,没补推给正在代看的负责人。
- **aitalky 规避**:代看场景下,成员会话的关键事件(入组、消息、翻译、设置变更)要**补推给正在代看的负责人**;会话设置变更要**双向实时同步**负责人与成员两端。
- 来源:cicada `8516a6d9`(入组补推)、`a79b1e38`(翻译补推)、`0e4cc0f5`(设置双向同步)、`799ac9de`(代管基础)。

### 坑 3.2 权限校验只认"本人账号" → 负责人代管报参数错误
- **现象**:负责人改成员的聊天模式等操作,因校验只认本人账号,报参数错误。
- **根因**:权限判断写死"操作对象 == 登录本人"。
- **aitalky 规避**:涉及代管的操作,权限判断用"**能否操作该账号**"(canOperateAccount:本租户内负责人可操作成员账号),而非"是不是本人"。对应 aitalky 的 RBAC + 多租户 project_id 隔离。
- 来源:cicada `8c666e85`(附带)、`f9722ddc`。

## 四、MongoDB / 持久化陷阱

### 坑 4.1 ⚠⚠ 空 Update 被 Mongo 当成"整文档替换" → 会话维护记录被清空、会话消失
- **现象**:标记已读接口,在消息已读时执行了一个**空的 Update**,Mongo 把空 `update` 当成整文档替换,把会话维护记录(`ConversationMaintainInfo`)清成只剩 `_id` —— 会话从列表消失、按 chatId 查不到,直到新消息 upsert 才恢复。**负责人+坐席同时标记已读必现**(第二次撞上 isRead=true 触发空 Update)。
- **根因**:`mongoTemplate.upsert(query, update)` 无条件执行,且 `update` 可能为空;空 Update 语义是"替换为空文档"。
- **aitalky 规避**:
  - 写库前**先判断是否真有变更**,无变更直接 return,不发空更新。
  - 局部字段更新用 `updateFirst` + 明确 `$set`/`$inc`,**不要 `upsert` 一个可能为空的 update**。
  - 顺带:`unreadCount` 等计数字段注意 **null 的 NPE**,递减前判空。
- 来源:cicada `2b9ed8b5`。**这是最隐蔽、后果最严重的一个,务必记住。**

### 坑 4.2 (对应 aitalky) MyBatis-Plus 主键 IdType.INPUT 漏 setId
> 这是 aitalky 自己踩的同类"空值写库"坑,一并记此:`BaseEntity` 主键 `IdType.INPUT` 不自生成,insert 前漏 `setId` 会运行时报 `Column 'id' cannot be null`。已在 `AuditMetaObjectHandler.insertFill` 兜底注入雪花ID根治。详见记忆 `baseentity-input-id-must-setid`。

## 五、单点登录 / Token / 多窗口

### 坑 5.1 token 存共享 localStorage → 同浏览器多窗口互相覆盖
- **现象**:一个客户开多个客服窗口,token/userId/userType 写在共享的 `localStorage`,后开的窗口覆盖先开的,导致 HTTP(发消息/拉历史)用错 token、会话串。
- **根因**:会话级凭证放进了浏览器**全局共享**的 localStorage。
- **aitalky 规避**:**会话级凭证(token/身份)用每标签独立的 `sessionStorage`**;只有 language 等真正的全局偏好才用 localStorage。aitalky 信使端(apps/messenger)尤其注意,客户可能多开。
- 来源:im-h5 `22ddafc`。

### 坑 5.2 被单点顶下线无任何提示
- **现象**:客户被新登录顶下线后,旧窗口静默失效,用户无感知。
- **aitalky 规避**:被顶下线时显示明确遮罩(如「当前窗口已失效」),不要让用户对着一个已死的窗口继续发消息。
- 来源:im-h5 `5553ac8`。

## 六、发送态与失败重发

### 坑 6.1 WS 未连通时 send 静默丢弃却照常渲染 → 用户误以为发成功
- **现象**:断网/WS 断开时 `ws.send()` 静默丢消息,前端照常把消息渲染成已发送,用户以为发出去了其实没有。
- **根因**:`send()` 不返回是否真正写入已连接 socket,前端无条件乐观渲染。
- **aitalky 规避**:
  - `send()` 要**返回是否真正写入已连接的 socket**;据此给消息打 `status`(success/failed/sending)。
  - 失败消息显著标记(红色感叹号)+ 点击重发;失败消息**不显示已读态**。
  - 重发成功后,用**重发时刻**作为新时间并把该消息**移到列表末尾**、滚动到底,与服务端/对方实际收到顺序一致;重发失败原地保持失败态。
- 来源:im-h5 `30feae4`、`66c8c11`。
- 注:这与 aitalky"WS 只当通知、seq 当真相"一致——**发送成功与否以服务端落库回执为准**,不靠前端乐观态。

## 七、头像 / 快照 / 展示一致性

### 坑 7.1 账号列表头像写死成登录用户头像 → 负责人代看时所有账号头像一样
- **现象**:客服渠道账号头像绑定为"当前登录用户"头像,负责人代看多账号时,列表里所有账号头像相同。
- **根因**:展示数据取了登录态而非数据本身。
- **aitalky 规避**:列表展示的头像/昵称等,**取数据所属对象自身的字段**(账号所属坐席的头像),不要图省事用当前登录用户的;无值再回退首字母头像。呼应 aitalky"发送者昵称/头像是发送时快照"的设计。
- 来源:cicada `57cb3e21`、cicada-web `29f84dd`。

## 八、开发自检清单(写相关模块前过一遍)

- [ ] **多坐席并存**:这个会话/消息/连接的寻址,是否带了 `accountId`(坐席)维度?只按 customerId/chatId 找,一定串台。
- [ ] **当前会话判断**:前端"是否插入当前窗口/当前账号"的所有分支,是否都同时比对了会话标识 + 账号标识?分支之间是否一致?
- [ ] **WS 漏帧**:重连/切前台是否会自动 `sync` 增量补拉?补回的消息是否**按 seq 有序插入**、覆盖整个缺口(逐页直到衔接)、统一幂等去重、对方消息照常提示?
- [ ] **水位线**:补拉基准用的是 **seq / 服务端时间戳**,绝不是本端本地时钟?
- [ ] **Mongo 更新**:有没有可能发出**空 Update**?是否会被当成整文档替换?无变更是否直接 return?计数字段递减前判空?
- [ ] **凭证存储**:会话级 token 是否用 `sessionStorage`(每标签独立)?多开窗口会不会互相覆盖?
- [ ] **发送态**:WS 没连通时发消息,是否如实标记 failed 并支持重发,而不是静默假装成功?
- [ ] **代管/代看**:负责人代看时,成员会话的实时事件是否补推?权限是否用"能否操作该账号"而非"是不是本人"?
- [ ] **展示取数**:头像/昵称等是否取自数据所属对象,而不是当前登录用户?
