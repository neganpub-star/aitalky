# 消息可靠投递与漏消息补偿（WS 断连/弱网丢消息）

> 背景:cicada 线上 bug —— 网络不好时 WS 断了,断连期间产生的消息没推到前端,这部分"漏消息"在前端永久缺失,直到用户手动刷新/重开会话才补回来(甚至刷新也不一定补,因为 cicada 把 WS 推送当成了消息流的唯一来源)。
> 本文给出 aitalky 的设计:**WS 只当"通知",seq 当"真相",三道补偿网兜底**。同时说明它如何与既有的"多端推送"设计协同。

---

## 0. 一句话结论

**永远不要把 WS 当作可靠投递通道,也不要用 WS 推流去"累积"完整消息历史。**
WS 是 best-effort 的通知;消息的完整性由「**会话内单调递增 seq + `sync(afterSeq)` 增量拉取 + 会话表权威字段**」保证。任何一环(弱网、ws 实例重启、RocketMQ 消费失败、跨实例未送达)丢了消息,都能被 seq 对账补回来。这把"可靠性"从**推送链路**解耦到了**拉取对账**——链路里任何一环丢都不致命。

---

## 1. 我们已有的地基（关键,补偿方案全靠它们）

| 能力 | 实现位置 | 说明 |
|---|---|---|
| 会话内单调 seq | `MessageServiceImpl.send` → `redisson.getAtomicLong("conv:seq:{cid}").incrementAndGet()` | 每条消息在**会话内**有唯一递增 seq(跨实例不重复) |
| 增量拉取 | `MessageService.sync(cid, afterSeq)` → `findByConversationIdAndSeqGreaterThanOrderBySeqAsc` | 拉 `seq > afterSeq` 的全部消息,升序;**返回的是实际落库的消息,自然跳过空洞** |
| 最近 N 条 | `loadLatest(cid, 50)` | 打开会话首屏 |
| 会话权威字段 | `cnv_conversation`:`last_seq` / `unread_count` / `last_message_preview` / `last_message_at` | DB 里的真相,列表/详情查询即可拿到,不依赖 WS |
| WS 推送带 seq | 推送 payload = `MessageVO`(含 `seq` / `msgId` / `conversationId`) | 客户端可据此做顺序/去重判断 |

> 核心:`cnv_conversation.last_seq` 是「这条会话当前最大 seq」的**权威值**;客户端任何时候拿它和本地 `localMaxSeq` 比,就知道有没有漏。

---

## 2. 客户端模型:localMaxSeq + 三道补偿网

客户端对**每条会话**维护一个 `localMaxSeq`(本地已收到的最大 seq)。所有补偿都围绕它。

### 网 ①:重连补拉（reconnect resync）—— 主力兜底
WS `onopen`(首连或重连成功)后:
1. 对**当前打开/已订阅**的会话,调用 `sync(cid, afterSeq=localMaxSeq)`,把断连期间的消息补回来,逐条 append + 去重,推进 `localMaxSeq`。
2. **刷新一次会话列表**:列表项的 `unreadCount` / `lastMessagePreview` / `lastMessageAt` 以服务端 DB 为权威,一次拉取即可纠正断连期间所有会话的角标与预览。

> 覆盖:断连 → 重连这段时间窗内的所有丢失。这是最主要、最常见的场景。

### 网 ②:Gap 检测（in-band,连着也可能丢中间一帧）
即便 WS 连着,也可能丢单帧。收到 WS 消息 `m` 时按 `m.seq` 判断:
- `m.seq == localMaxSeq + 1` → 正常,append,`localMaxSeq = m.seq`。
- `m.seq <= localMaxSeq` → 重复/旧帧(补拉与推送交叉、自己发的回声),**丢弃去重**。
- `m.seq > localMaxSeq + 1` → **中间有空洞** → 触发 `sync(cid, afterSeq=localMaxSeq)` 补齐,再把 `m` 接上(sync 通常已经把 m 一并带回,靠去重收敛)。

> 覆盖:连接正常但中途漏帧。下一帧到达即自愈。

### 网 ③:兜底轮询 / 窗口聚焦补偿（最后一帧也丢的极端情况）
若**最后一条**消息丢了,且后面没有新帧来触发网②,客户端无从感知。兜底:
- **会话列表轮询**(已实现,10s):列表查询返回各会话的 `unreadCount` / `lastMessagePreview`(DB 权威)。当前会话若发现服务端有更新而本地没有 → 对当前会话 `sync` 一次。
- **窗口 `focus` 时**:对当前打开会话 `sync(afterSeq=localMaxSeq)` 一次(用户切回标签页/唤醒电脑,常伴随断连)。

> 覆盖:静默期最后一帧丢失。代价:最坏延迟 = 轮询周期(10s)。

## 3. ⚠ 必须注意的坑:seq 空洞不可填 → gap 检测不能"死等连续"

**现状**:`send()` 里 seq 是 `incrementAndGet()` **先分配、后落库**。若落库(Mongo save)失败,这个 seq 就被永久浪费 → 会话 seq 序列出现**永久空洞**(如 …5,6,8,9…,7 不存在)。

**若客户端 gap 检测死等 `localMaxSeq+1`,会在空洞处永久卡住**(反复 sync 都补不到那个 seq,后面的消息再也接不上)。

**规则(铁律):**
> `localMaxSeq` 一律以「**已落库消息的实际 seq**」前进,**绝不按 `+1` 推断连续**。
> - `sync` 返回后,用返回集合里的 **实际 max(seq)** 更新 `localMaxSeq`,哪怕中间仍不连续也接受(空洞被自然跳过)。
> - 网② 的"gap 检测"只用来**触发一次 sync**,不用来阻塞推进;sync 回来即便仍有洞,也照常前进。
> - 一句话:**用"单调性"推进与去重,不要用"连续性"**。

**根治(可选,二期):** 让 seq 分配与落库一致 —— 落库成功后才认 seq(失败回滚/重试),或把 seq 写进消息后用唯一约束保证;即便不根治,只要客户端容忍空洞就不会卡。

## 4. 与「多端推送」的协同（呼应既有设计）

既有多端设计(见 `PushService` / `ConnectionRegistry`):一个成员的**全部连接**(多标签/Web+App)都登记在 `ws:conn:member:{id}`,推送给全部连接;负责人代看靠会话订阅集合 `ws:sub:{cid}` 覆盖;引用计数在线、**不做单点登录顶号**。漏消息补偿要在多端下保持一致:

1. **每个连接各自对账**:每个端/标签页各自维护自己的 `localMaxSeq`,各自跑三道网,互不干扰。某个端弱网漏了,只补它自己。
2. **未读数以服务端为权威,不靠各端本地累加**:
   - 本地累加(收到客户消息 `unread+1`)只为**即时体验**;
   - **刷新/重连/轮询一律以 `cnv_conversation.unread_count` 为准**覆盖本地值。否则多端各算各的(cicada 的老毛病之一)。
   - 打开会话即 `resetUnread` 清零(已实现);所有端下次拉列表就看到一致的 0。
3. **多端"已读"精确同步需要服务端已读位点(二期)**:
   - 一期:打开会话即整会话 `unread=0`(够用,粗粒度)。
   - 二期:客户端上报「已读到 seq=X」,服务端存 per-member 的 `last_read_seq`;`unread = lastSeq - lastReadSeq`。这样 A 端读到一半,B 端的未读也精确同步,并支撑**已读回执**。

## 5. 链路任何一环丢都被兜住（为什么这套是对的）

消息从 `app 落库 → RocketMQ → ws 消费 → 下发连接` 链路上的丢失点,统统被 seq 对账兜住:

| 丢失点 | 是否落库 | 被哪道网补 |
|---|---|---|
| 弱网/客户端断连 | 是 | 网①重连补拉 / 网③ |
| WS 单帧丢失 | 是 | 网② gap 检测 |
| ws 实例重启、连接迁移 | 是 | 网①(重连必触发) |
| RocketMQ 消费失败/积压 | 是 | 网③轮询 + 下次任意帧触发网② |
| 跨实例推送(`PushService` 里 TODO 未接 MQ) | 是 | 网①/网③ —— **所以跨实例 MQ 没接也不致命,先靠对账兜着** |
| send 落库失败 | **否** | 不是消息丢,是消息没产生;seq 空洞由 §3 规则容忍 |

> 关键洞察:**只要消息落了库 + 有 seq,前端就一定能补回来**。推送链路退化为"加速器",不再是"正确性的唯一来源"。

## 7. 设计取舍

- **为什么不用 WS ack + 服务端重发队列(QoS1)?** 那是 IM 长连接网关的重武器(服务端为每连接维护未确认队列 + 超时重推),实现复杂、状态重、横向扩展难。我们已有 seq + DB 权威,用"客户端对账拉取"以极低成本达到等价的最终一致,且对跨实例/MQ 故障天然鲁棒。**对客服场景(消息量中等、容忍秒级补偿延迟)是更优解。**
- **轮询会不会浪费?** 列表轮询本就是新会话进列表的兜底(WS 只覆盖已订阅会话),顺带承担网③,边际成本几乎为零。
- **延迟保证?** 正常路径(WS 通)≈ 实时;丢消息后:网①重连即时、网②下一帧即时、网③最坏一个轮询周期(10s)。

## 8. 体验优化:全自动无感补偿（对标 cicada「要手动刷新」）

> **cicada 的真实问题**:消息其实已落 MongoDB,**没丢**;但前端把 WS 推送当成消息流的唯一增量来源,**重连后不会自动对账拉取**,只有用户**整页刷新 / 重开会话**(重新 `loadLatest`)才把库里的漏消息显示出来。本质是「前端对账没有自动化」。
> **我们的目标**:漏消息**后台无感自动补上**,用户**永远不需要刷新**;最坏可见延迟做到秒级。

### 8.1 引入 `serverLastSeq` 对账(比 gap 检测更主动)
让**会话的当前最大 seq(`cnv_conversation.last_seq`)随各种响应回带给前端**,前端每次拿到就和本地 `localMaxSeq` 比:
> `serverLastSeq > localMaxSeq` ⇒ 我落后了 ⇒ **自动 `sync` 补**。

回带 `serverLastSeq` 的载体(越多越早发现):
- **列表轮询 / 重连刷列表**:`ConversationVO` 增加 `lastSeq` 字段 → 每个会话都能对账(当前会话自动补消息,其他会话自动纠未读角标)。
- **打开会话详情**:`ConversationDetailVO` 增加 `lastSeq` → 作为该会话对账基准。
- **(可选,二期)WS 心跳 `pong` 夹带**:服务端在 `pong` 里回带「该连接订阅的会话们的 last_seq」→ 不依赖新消息、不依赖轮询,**心跳周期内必然发现落后**,这是最主动的探针。

> 对比:网②(gap 检测)只能在「**下一帧到达**」时发现漏;`serverLastSeq` 对账能在「**任意一次轮询/心跳/别的会话推送**」时发现漏 —— 连「静默期最后一帧丢失」也不用等 focus。

### 8.2 当前会话「快补」:轮询分级
- **当前打开的会话**:对账周期调短(如 3~5s)或直接靠 WS `pong` 夹带 → 准实时补。
- **其他会话(仅角标/预览)**:维持 10s 列表轮询即可,成本低。

### 8.3 补回来的消息要「按 seq 有序插入」,不是 append 到底
漏的可能是**中间**的消息(不只是末尾)。前端消息列表以 **seq 为序维护**(合并去重后按 seq 排序插入),保证补拉回来的旧 seq 插到正确位置。
> cicada 刷新是「整体重拉」所以顺序天然对;我们是**增量插入**,必须自己保证顺序 —— 这是增量方案的代价,但换来了无感。

### 8.4 未读 / 列表预览 一起自动补(不只聊天区)
自动对账不仅补「当前会话的聊天消息」,也要把**列表的未读数、最后一条预览**一起纠正(以 DB 权威值覆盖本地)。否则会出现「聊天区补上了,但列表角标还是错的」的割裂。

### 8.5 最终效果
| 场景 | cicada | aitalky |
|---|---|---|
| 断连重连后的漏消息 | 要手动刷新才显示 | onopen 自动 sync,无感补上 |
| 连接正常丢中间一帧 | 可能一直缺到刷新 | 下一帧 gap 检测 / serverLastSeq 对账自动补 |
| 静默期最后一帧丢失 | 要手动刷新 | 轮询/心跳 serverLastSeq 对账自动补(秒级) |
| 漏的是中间消息 | 刷新后整体重排 | 按 seq 有序插入到正确位置 |
| 多端未读不一致 | 各端各算 | DB `unread_count` 权威,自动对齐 |

> 一句话:**把 cicada「被动等刷新」改成「主动持续对账」,触发源从单一(刷新)扩展到多路(重连/轮询/心跳/gap/聚焦),全程后台无感。**
