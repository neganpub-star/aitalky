# ByteTrack 参考分析（V1 重写蓝图）

> 用途：领导要求新客服系统**参考 ByteTrack**（bytrack.com）。本文记录 ByteTrack 的产品概念、完整使用流程、关键设计点，并对照现有 cicada，给出**重写版 V1 的领域模型与设计建议**。
>
> 资料来源：ByteTrack 官网 + 帮助中心《ByteTrack 使用指南》正文 + 控制台截图。截图存于 `./images/`（编号对应下文）。
> 整理时间：2026-06-08。

## 一、ByteTrack 是什么

- **定位**：面向 **Web3 行业**（加密交易所、DeFi、NFT）的 **AGI 智能客服 + 运营平台**。
- **模块**：智能客服（在线客服/实时翻译/多渠道/快捷回复/会话洞察）、智能流程、数字员工(AI agent)、知识库、通知推送、数据指标（客户生命周期价值、情绪价值指数 EVI、NPS、CES）。
- **接入**：非侵入式插件，端到端加密，支持独立部署。
- **本质**：Intercom 那一类"嵌入式网页客服(Messenger) + 共享收件箱 + 会话分配"产品，叠加 AGI 能力。

> 关键认知：**现有 cicada 本就是照着 ByteTrack 概念建的**（信使/项目/会话/信使设置 一一对应）。"参考 ByteTrack" = 把这套概念**干净重做**。

## 三、ByteTrack 完整使用流程（《使用指南》正文提炼）

1. **注册账号** — 邮箱注册（邮箱即账号，用于收验证码）。`images/01-register.png`
2. **登录系统** — 账号密码 + 邮箱验证码（2FA）。`02-login-code.png` `03-login.png`
3. **创建项目** — 一账号可建多个项目；首次登录引导建第一个项目（如"ZARA专卖店"）。`04-create-project.png`
4. **进入项目** — 左上角显示当前项目名，项目列表可切换。`05-enter-project.png`
5. **订阅套餐** — 项目需订阅才能用主要服务；套餐**按项目**生效。`06-subscribe.png`
6. **组建团队** — 邀请链接邀人加入项目；团队设置→成员管理（改昵称/头像、禁用、删除）。`07~11`
7. **分配权限** — 团队设置→权限管理；建角色(如"客服")→只勾"收件箱"模块→成员管理里给成员配角色 ⇒ **客服之间互相看不到会话**。`12-role-create.png` `13-role-permission.png` `14-member-assign-role.png` `15-agent-own-conversations.png`
8. **设置客服组（分配模式）** — 设置→信使设置→会话设置。`16-conversation-setting.png`
   - **8.1 普通分配模式**：共享队列，成员加入队列，有消息时按规则分配并成为负责人。`17-normal-match-queue.png`
   - **8.2 专属分配模式**：多业务入口(售前/售后…)，每个**专属客服组**有独立接入URL+队友，按入口路由。`18-exclusive-match-group.png`
9. **安装信使** — 三种接入：**URL** / **网页插件(JS)** / **SDK**。`19-url-access.png`
   - URL 参数：`appId`、`groupId`、`lang`(16语言)、`redirectUrl`(关闭返回业务页)、`userId`(业务UID)。`20~24`
10. **最佳实践** — 保持在线(左下角切工作状态 `25`)、及时关闭会话(`26`)、设置坐席最大会话承载量(`27`)。

## 四、关键设计点（V1 必须吃透的）

### 1) 会话分配模型（与 cicada 最大的不同）
ByteTrack 是**共享收件箱 + 分配**，不是 cicada 的"客户直连某账号"。`images/26` 收件箱左栏：**我的 / 提及我的 / 未分配 / 全部**。

- **分配规则**（`16`）：
  - **手动分配**：不自动分配，进未分配池等人接。
  - **轮流分配 (round-robin)**：按顺序轮流分给队友。
  - **负载分配**：分给"在线且进行中会话最少"的队友。
- **分配限制**：每坐席**最大并发会话数**（只算"进行中"）；达上限不再参与自动分配。（手动分配/专属模式不受此限）
- **普通模式**：一个共享队列（加入队列的成员参与自动分配）。
- **专属模式**（`18`）：多个命名客服组（如"售前业务线""售后业务线"），**每组一个接入URL**，从该URL进来的会话只分给该组队友。用于多业务线/多入口。

> ✅ **V1 直接采用这个模型**：会话进池 → 按规则分配给坐席(assignee) → 可转派/结束。这样"负责人看全部、坐席看自己"是**权限+分配的自然结果**，不用再像 cicada 那样硬凑"代看"。

### 2) 权限隔离（RBAC）
角色按**模块**授权。"客服互相看不到会话" = 给"客服"角色只开"收件箱" + 分配模型决定每人只看分给自己的会话。负责人/管理员角色开全部模块 ⇒ 天然能看全部。
> ✅ 取代 cicada 这周做的"负责人代看"补丁，用标准 RBAC + 分配可见性。

### 3) 信使接入（V1 的唯一渠道）
- **URL 接入**最实用：`http://<host>/?appId=xxx&lang=zh_CN&userId=abc123&redirectUrl=...&groupId=g44DfQ`
  - `userId`：业务系统用户唯一标识，**按它聚合该用户的所有会话**。⚠️ **安全**：必须复杂无序（如"账号-注册时间"），否则被猜到会泄露他人会话（cicada 的客户身份正是这个思路）。
  - `redirectUrl`：带则信使有"关闭"按钮、点了回跳业务页（需 urlencode）。
  - `groupId`：指定专属客服组（多入口）。
  - `lang`：16 种语言(zh_CN/en_US/zh_TW/ja_JP/ko_KR/vi_VN/ms_MY/id_ID/ru_RU/th_TH/mm_MM/la_LA/fr_FR/it_IT/es_ES/de_DE)。
- **窗口重定向**：建议业务方用 `window.open(url, windowName)` 复用窗口，避免重复开窗。（cicada 这周踩的"多窗口串消息"正源于此——V1 设计客户端单例/复用窗口）
- JS 插件 / SDK 接入：V1 可只做 URL + JS 插件，SDK 后置。

### 4) 会话生命周期 + 协作
- 状态：进行中 / 已结束；坐席处理完应**及时关闭**（影响承载量计数与统计）。
- **内部备注(internal note)** + **@提及**：坐席间私下协作，客户不可见（cicada 没有，V1 应加）。
- **会话资料面板**：会话ID/IP/地点/业务UID/邮箱/联系方式等客户画像。

### 5) 富消息
文本、图片、链接、**文章/商品卡片**（结构化字段）、知识库内容。V1 至少支持 文本/图片/文件，卡片可后置。

## 五、与 cicada 现状对比（为什么重写更干净）

| 维度 | cicada 现状(技术债) | ByteTrack / V1 目标 |
|---|---|---|
| 会话归属 | 客户直连某坐席账号；多坐席靠"负责人代看"补丁 | **共享收件箱 + 分配/转派**，RBAC 控可见性 |
| 用户模型 | im_user / sys_user 双表，昵称来源不一致（本周 bug 根源） | **单一用户模型** |
| 渠道 | TG/FB/TikTok/Twitter/企微 一大堆（多数用不上） | **只做 web 信使**，渠道抽象成插件 |
| 框架 | RuoYi-Vue-Plus 重框架 + 前人硬整合，业务散落 | 干净**模块化单体** + DDD 边界 |
| MQ | RabbitMQ + RocketMQ 两套 | **一套** |
| 死代码 | forwardToSupervisor 等从未调用 | YAGNI，不留废线 |
| 前端 | Vue2(工作台) + Vue3(后台) + Vue3(h5) 混用 | **统一 Vue3 + TS + Vite** |
| 协作 | 无内部备注/@/转派 | 有 |

## 六、V1 重写建议（结论）

**范围**：客服核心（信使接入 + 共享收件箱/分配 + 会话/消息 + 团队/RBAC + 项目/订阅）+ **AI 翻译**。
**不做(V2)**：知识库、数字员工(agent)、流程自动化、数据洞察指标、AI 自动回复。

**技术栈**：
- 后端：**Java 21 + Spring Boot 3.x**（团队强项），**扔掉 RuoYi**，自建模块化单体 + DDD。
- MQ 一套（RabbitMQ 足够）；存储考虑 PostgreSQL 一库(JSONB 存消息) 或 保留 Mongo 但统一用户模型；Redis(缓存/会话/WS路由)；MinIO(文件)。
- WS：Netty 或 Spring WebSocket。
- 前端：Vue3 + TS + Vite + Pinia，三端(客户信使/坐席台/后台)一套栈。
- AI 翻译：Java 内一个模块调翻译 API；**预留"智能"边界上下文**，V2 接知识库/agent 时往里加，不动核心。

**领域模块划分(草案)**：
1. `身份`：账号、登录(邮箱验证码)、项目、项目成员、角色权限(RBAC)
2. `信使`：接入(URL/JS)、信使设置、客户身份(userId 聚合)、多语言
3. `会话`：会话生命周期、**分配引擎**(手动/轮流/负载 + 承载量 + 普通/专属组)、转派、内部备注、@提及、客户画像
4. `消息`：收发、富消息、已读回执、撤回、实时 WS、**翻译**
5. `坐席工作台`：收件箱视图(我的/未分配/全部)、工作状态
6. `订阅计费`：套餐(按项目)
7. `智能(预留)`：V1 仅翻译；V2 知识库/agent/流程

**从 cicada 搬运的"血泪经验"（变成 V1 需求）**：
- 客户端**单例/窗口复用**，避免多窗口串消息（本周 bug）。
- 客户身份(userId)聚合会话 + **复杂无序防猜测**。
- 实时三件套要从一开始设计干净：消息推送、已读回执、撤回。
- 会话与坐席的关系用**分配(assignee)**建模，不要用"账号绑定"。
